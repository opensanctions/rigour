# Rust-port sequence: the name tagger

Companion plan to:
- `plans/rust.md` — umbrella port; this is Phase 4.
- `plans/rust-resources.md` — JSON shapes for the data that moves.
- `plans/rust-symbols.md` — Symbol representation + interner.
- `plans/rust-names-parser.md` — `analyze_names` pipeline that consumes
  tagger output (Phase 5, after this).
- `plans/rust-normalizer.md` — `normalize_flags=` / `cleanup=`
  parameter pattern the tagger entry points adopt.

## Context

Today `rigour/names/tagging.py` builds two Aho-Corasick automata
(`_get_org_tagger`, `_get_person_tagger`) from six YAML-sourced
wordlists plus the Wikidata-derived person-names corpus, and exposes
`tag_org_name` / `tag_person_name` that apply those automata to
`Name` objects. It's the main performance hot spot in the matching
stack.

The Rust port is a bottom-up migration: ship Symbol first, then move
each resource file one by one, then assemble the tagger. Each step is
independently reviewable and most are independently deployable.

## Data classification

A useful distinction for sizing per-step effort: which of these
resources is still read by Python after the tagger port, and which
becomes Rust-only?

| Resource | Post-port consumer | PyO3 accessor needed? |
|---|---|---|
| `resources/text/stopwords.yml` — `STOPWORDS`, `NULLWORDS`, `NULLPLACES` | Python — `rigour/text/stopwords.py` still implements `is_stopword` / `is_nullword` / `is_nullplace` | **Yes** |
| `resources/names/stopwords.yml` — `PERSON_NAME_PREFIXES`, `ORG_NAME_PREFIXES`, `OBJ_NAME_PREFIXES`, `NAME_SPLIT_PHRASES`, `GENERIC_PERSON_NAMES` | Python — `rigour/names/{prefix,split_phrases,check}.py` | **Yes** |
| `resources/text/ordinals.yml` | Python (`rigour/addresses/normalize.py`) and Rust (tagger build) | **Yes** for the Python side; Rust tagger `include_str!`s the same JSON internally |
| `resources/names/symbols.yml` — `org_symbols`, `org_domains`, `person_symbols`, `person_nick`, `person_name_parts` | Rust-only — only the tagger consumes these | **No** |
| Territory name aliases (stripped subset from `resources/territories/*.yml`) | Rust-only — tagger LOCATION symbols | **No** |
| `persons.txt` (committed plain; `build.rs` compresses to OUT_DIR) | Rust-only — person tagger | **No** |

The Rust-only artifacts still live under `rust/data/…`, still get
regenerated by `make rust-data`, still get covered by the CI no-diff
check. They just have no `#[pyfunction]` wrapping them — the tagger
reads them inside the Rust crate via `include_str!` / `include_bytes!`.
No `.pyi` stub, no Python-side accessor, no export from
`rigour._core`.

## What falls out at each step

- **Steps 5, 6, 7** are pure data moves — no `.pyi`, no `_core`
  function, no Python consumer rewired. Smallest possible PR each.
  They can land before step 8 (where they become inert, committed
  JSON/blobs in `rust/data/`) or alongside it. Either is fine.
- **Steps 2, 3, 4** rewire Python consumers to read from Rust, which
  retires `rigour/data/**/*.py` files and produces an immediate
  testable delta. These are the "real" data-migration PRs.
- **Step 8** is the load-bearing port. It consumes everything from
  steps 1–7 and replaces the whole of `rigour/names/tagging.py`.

    Org,
    Person,

    needles: Needles<Vec<Symbol>>,

impl Tagger {
    /// Apply tagger to pre-normalised text, yield (matched, symbol)
    /// pairs — one entry per (match, symbol in payload) pair, matching
    /// today's Python `Tagger.__call__` output shape.
    pub fn tag<'a>(&'a self, text: &'a str) -> Vec<(&'a str, Symbol)> {
        let mut out = Vec::new();
        for m in self.needles.find_iter(text) {
            for sym in m.payload {
                out.push((m.matched, sym.clone()));
            }
        }
        out
    }
}

Each is a `LazyLock` that pays its load cost once per process. The
persons corpus is the big one — ~8.5 MB of plain UTF-8 committed at
`rust/data/names/persons.txt`, compressed to ~2.7 MB by `build.rs`
into `OUT_DIR` at crate-build time, and decompressed once on first
access via `names::persons::raw()`. The tagger build then pays
another ~50–200 ms for parsing + AC construction on top.

## Symbol payload: `Vec<Symbol>` vs `SmallVec<[Symbol; N]>`

The today's Python tagger maps phrase → `Set[Symbol]`. Most phrases
yield 1 symbol; some yield 2–3 (ordinals that appear in both org and
person taggers, or aliases that map to an org type AND a domain).

`Vec<Symbol>` is the obvious choice. `SmallVec<[Symbol; 2]>` or
`SmallVec<[Symbol; 4]>` would save a heap allocation on the common
case (Symbol is 24 B, so 2–4 inline is cheap). Defer this as a
profiling-driven tweak.

`HashSet<Symbol>` (matching today's Python) is also possible but
adds a hash table per phrase, which is wasteful at this count.
Deduplicate during `mapping.entry(...).or_default().push(sym)` by
walking the Vec linearly — for N ≤ 4 it's faster than a HashSet.

## Verification

### Per-step (2–7)

- `make rust-data` produces the new JSON/blob, committed artifact
  matches bit-for-bit between runs.
- CI no-diff check passes.
- Python consumer tests continue to pass unchanged (for steps 2–4).
- Steps 5–7 have no Python-observable effect; sanity-check via
  `cargo test` (the tagger's unit tests pick them up once step 8
  lands; before that, just `cargo build`).

### Step 8 (tagger itself)

- Rust unit tests in `rust/src/names/tagger.rs` covering: ordinal
  matching, org-symbol matching, person-symbol matching, word-boundary
  correctness (inherited from `Needles<T>`), cache hit/miss per
  flag combo.
- Python parity tests in `tests/names/test_tagging.py` — the existing
  tests should pass by adjusting the `tag_org_name(name, normalizer)`
  calls to `tag_org_name(name, normalize_flags=Normalize.CASEFOLD)`
  etc. Behaviourally identical output.
- `cargo clippy --all-targets -- -D warnings` clean.
- `mypy --strict rigour` clean.
- Benchmark: run a representative matching workload (e.g. 1k entities
  × 10 candidates) and compare Python-tagger vs. Rust-tagger throughput.
  Expect a meaningful improvement given org_types at 190×.

## Out of scope

- **Porting `is_stopword` / `is_nullword` / `is_generic_person_name`
  to Rust.** They consume the same data as the tagger (via step 2/3
  JSON), but they're standalone functions that benefit less from
  FFI reduction. Can follow the org_types pattern later if profiling
  shows they're hot.
- **Splitting `rigour/names/tagging.py` between `Tagger` class and
  the `tag_*_name` entry points.** The Rust port collapses both into
  `rust/src/names/tagger.rs`; the Python `Tagger` class disappears.
  Downstream consumers don't import `Tagger` directly.
- **NamePart / Span / Name ports.** Step 8 still takes a Python-owned
  `Name` and mutates it in place. The full object-graph port is
  `rust.md` Phase 2, which runs in parallel with this sequence and
  lands independently.
- **`ahocorasick-rs` dep removal.** Happens as a consequence of step
  8, not a separate PR — once nothing imports it, drop from
  `pyproject.toml`.
